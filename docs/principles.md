# Wiscord — Guiding Principles

## Single product test

Every feature must answer one question:

> **Does this make studying with others feel better than studying alone?**

If not, cut it.

## Scope discipline

When in doubt, build the smallest version that works. Push back on requests that drift into the phase 2 list (quizzes, analytics, streaks, proactive AI, summaries, etc. — see [overview.md](./overview.md)).

## Feel target

> **"Linear with a heartbeat, not a cyberpunk dashboard."**

Fast and clean over flashy. The timer, presence, and AI should work together so studying with others feels productive, not lonely.

## Why

Small team, one-month v1. A focused, shippable product beats a feature-complete clone of Discord + Notion + study-app. Every cut feature is time bought back for polish on the core loop.

## How to apply

When evaluating any proposal — new feature, abstraction, library, design flourish — run it through the single product test first, then the smallest-version-that-works test. If a request sounds like phase 2, name the conflict explicitly and offer the v1-shaped alternative.
